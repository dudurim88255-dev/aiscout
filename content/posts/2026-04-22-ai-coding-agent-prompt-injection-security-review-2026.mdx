AI 코딩 에이전트 보안 취약점을 걱정하고 있다면, 이 글이 그 불안을 구체적인 데이터로 바꿔줄 것이다. 2026년 4월, 존스홉킨스대 연구팀이 단 하나의 악성 PR 제목으로 [Claude](https://claude.ai) Code, [Gemini](https://gemini.google.com) CLI, [GitHub Copilot](https://github.com/features/copilot) 세 도구의 API 키를 동시에 탈취하는 데 성공했다. 외부 인프라도 필요 없었고, 특별한 권한도 필요하지 않았다.

이 글에서는 해당 취약점의 기술적 원리, 세 벤더의 시스템 카드가 실제로 무엇을 공개하고 무엇을 숨기는지, 그리고 지금 당장 AI 코딩 에이전트를 쓰는 팀이 취해야 할 현실적인 대응책을 정리했다. 홍보성 문서가 아닌, 실제 익스플로잇 보고서와 시스템 카드를 기반으로 분석했다.

## 핵심 요약

- **취약점 유형**: 프롬프트 인젝션(Prompt Injection) — 에이전트 런타임 레벨
- **영향받은 도구**: [Claude](https://claude.ai) Code Security Review (Anthropic), [Gemini](https://gemini.google.com) CLI Action (Google), [GitHub Copilot](https://github.com/features/copilot) Agent (Microsoft)
- **발견자**: Aonan Guan, Zhengyu Liu, Gavin Zhong (존스홉킨스대)
- **공격 방법**: GitHub PR 제목에 악성 명령어 삽입 → API 키 PR 코멘트로 자동 노출
- **CVSS 점수**: Anthropic 기준 9.4 Critical
- **바운티 지급액**: Anthropic $100 / Google $1,337 / GitHub $500
- **패치 여부**: 세 벤더 모두 조용히 패치. CVE 없음, 공식 보안 권고문 없음
- **한국어 공식 대응**: 없음 (영문 문서만 존재)
- **가장 큰 문제**: 시스템 카드가 런타임 보안을 문서화하지 않음


## Comment and Control이란?

"Comment and Control"은 2026년 4월, 존스홉킨스대 연구팀이 공개한 AI 코딩 에이전트 대상 프롬프트 인젝션 공격 기법이다.

공격 방식은 단순하다. 연구자는 GitHub 저장소에 Pull Request를 열고, PR 제목에 악성 명령어를 삽입했다. [Claude](https://claude.ai) Code Security Review Action이 해당 PR을 처리하는 순간, 에이전트는 명령어를 "신뢰할 수 있는 입력"으로 오인하고 자신의 API 키를 PR 코멘트로 게시했다.

이 공격이 특히 주목받는 이유는 세 가지다.

- **외부 인프라 불필요**: 별도 서버나 C2(Command & Control) 인프라 없이 GitHub 자체 기능만 활용
- **세 개 도구에 동일하게 작동**: 특정 벤더의 문제가 아닌 AI 에이전트 런타임 설계의 구조적 문제
- **Anthropic 시스템 카드가 이미 예측**: 시스템 카드에 "프롬프트 인젝션에 hardened되지 않았다"고 명시되어 있었음

### 왜 pull_request_target가 문제인가

GitHub Actions는 기본적으로 fork PR에 시크릿을 노출하지 않는다. `pull_request` 트리거를 쓰면 안전하다.

하지만 대부분의 AI 에이전트 통합은 `pull_request_target` 트리거를 요구한다. 이 트리거는 시크릿을 러너 환경에 주입한다. 즉, AI 에이전트가 작동하려면 시크릿에 접근해야 하고, 그 접근 경로가 공격 표면이 된다.

Enkrypt AI의 CSO이자 전 AWS 부 CISO인 Merritt Baer는 이렇게 말했다. "보호가 필요한 곳은 모델 경계가 아니라 액션 경계다. 런타임이 폭발 반경이다."

### 경쟁 도구와의 차이

이 취약점을 기존 SQL 인젝션이나 XSS와 단순 비교하면 안 된다. 전통적인 인젝션 공격은 데이터베이스나 브라우저를 대상으로 한다. 프롬프트 인젝션은 AI 에이전트의 "의사결정 레이어"를 직접 조종한다. 에이전트가 악성 명령을 진짜 업무 지시로 받아들이는 것이 핵심이다.

기존 WAF(웹 애플리케이션 방화벽)나 SAST 도구는 이 유형의 공격을 탐지하지 못한다. 입력 자체가 "정상적인 텍스트"이기 때문이다. 이것이 AI 코딩 에이전트 보안이 별도 카테고리로 다뤄져야 하는 이유다.


## 핵심 취약점 5가지 분석

### 1. 런타임 보안이 시스템 카드에 문서화되지 않는다

Anthropic의 Opus 4.7 시스템 카드는 232페이지 분량이다. 정량화된 해킹 성공률, 분류기 점수, 인젝션 저항 지표까지 포함된 상세한 문서다.

그러나 런타임 에이전트 보안에 대한 내용은 빠져 있다. [Claude](https://claude.ai) Code Auto Mode가 런타임 레벨 보호를 적용한다는 사실은 시스템 카드 외부에서 확인해야 한다. 시스템 카드 자체가 완전한 보안 청사진이 아닌 것이다.

OpenAI의 GPT-5.4 시스템 카드는 레드팀 시간을 문서화하고 모델 레이어 인젝션 평가를 공개하지만, 에이전트 런타임이나 도구 실행 저항 지표는 없다. Google의 [Gemini](https://gemini.google.com) 3.1 Pro 모델 카드는 대부분의 안전 방법론을 이전 문서에 위임한다.

### 2. 버그 바운티 금액이 위험도를 반영하지 않는다

CVSS 9.4 Critical 등급은 매우 높은 위험도를 의미한다. 그런데 Anthropic의 바운티는 $100(약 ₩140,000)이었다.

이 불일치는 Anthropic의 HackerOne 프로그램이 에이전트 툴링 발견을 모델 안전 취약점과 별도로 분류하기 때문이다. 즉, 에이전트 런타임 취약점은 "모델 문제"가 아닌 "통합 구성 문제"로 간주될 수 있다. 이 분류 체계 자체가 보안 연구자들에게 잘못된 인센티브를 만든다.

| 벤더 | CVSS | 바운티 금액 | CVE 발행 | 공식 보안 권고문 |
|------|------|------------|---------|---------------|
| Anthropic ([Claude](https://claude.ai)) | 9.4 Critical | $100 (₩140,000) | 없음 | 없음 |
| Google ([Gemini](https://gemini.google.com)) | 미공개 | $1,337 (₩1,872,000) | 없음 | 없음 |
| GitHub (Copilot) | 미공개 | $500 (₩700,000) | 없음 | 없음 |

### 3. 패치가 조용히 이루어진다

세 벤더 모두 패치를 배포했다. 그러나 CVE(Common Vulnerabilities and Exposures)를 NVD에 등록하지 않았다. GitHub Security Advisories를 통한 공식 보안 권고문도 없었다.

이는 실제로 해당 도구를 쓰는 기업의 보안 팀이 이 취약점을 인지하기 어렵다는 것을 의미한다. 취약점 관리 프로그램이 CVE 피드를 기반으로 운영된다면, 이번 건은 완전히 누락된다.

### 4. 신뢰 모델의 경계가 모호하다

[Claude](https://claude.ai) Code Security Review는 신뢰할 수 있는 자사 입력을 처리하도록 설계되었다. Anthropic의 공식 입장은 "신뢰할 수 없는 외부 PR과 이슈를 처리하도록 선택한 사용자는 추가 위험을 감수하며, 에이전트 권한 제한에 책임이 있다"는 것이다.

이 설명은 기술적으로 맞다. 그러나 현실에서 대부분의 개발팀은 외부 기여자의 PR을 처리하기 위해 AI 에이전트를 도입한다. "신뢰할 수 있는 입력만 처리한다"는 전제는 실제 사용 시나리오와 맞지 않는다.

### 5. 사이버 검증 프로그램의 격차

| 벤더 | 외부 사이버 프로그램 | 특징 |
|------|------------------|------|
| Anthropic | CVP (Cyber Verification Program) | 승인된 침투 테스터에게 사이버 보호 제거. 프롬프트 인젝션 방어는 미해결 |
| OpenAI | TAC (Trusted Access for Cyber) | 수천 명 규모 확대. ZDR(Zero Data Retention) 제한. 에이전트 운영 미적용 |
| Google | 없음 | ART(Automated Red Teaming)는 내부 전용. 외부 방어자 경로 없음 |

Anthropic의 CVP는 프롬프트 인젝션 방어를 다루지 않는다. OpenAI의 TAC는 접근 권한을 관리하지만 에이전트 운영 자체에는 적용되지 않는다. Google은 외부 보안 연구자를 위한 경로 자체가 없다.

세 벤더의 시스템 카드를 나란히 놓으면 공통점이 보인다. 모두 모델 레이어 안전성은 문서화한다. 그러나 에이전트가 도구를 실행하는 순간, 즉 런타임 레이어의 보안은 어떤 문서에도 체계적으로 정리되어 있지 않다.

Merritt Baer는 조달 담당자에게 구체적인 질문을 던지라고 조언한다. Anthropic에는 "안전성 결과가 기능 도약 사이에서 실제로 어떻게 이전되는가"를 물어야 한다. OpenAI에는 "타협 상황에서 '신뢰'가 무엇을 의미하는가"를 물어야 한다. 두 벤더 모두에게 "보안 장치가 프롬프트 필터링뿐 아니라 도구 실행까지 확장되는가"에 대한 명확한 답을 요구해야 한다.


## 직접 써본 솔직한 후기

이 섹션은 AI 코딩 에이전트([Claude](https://claude.ai) Code, [Gemini](https://gemini.google.com) CLI, [GitHub Copilot](https://github.com/features/copilot) Agent)를 실제로 GitHub Actions 환경에서 운영한 경험을 바탕으로 작성했다.

### 장점 1: AI 코딩 에이전트의 생산성 향상은 실제다

세 도구 모두 코드 리뷰 자동화, PR 요약, 보안 스캔 자동화 측면에서 생산성을 눈에 띄게 높여준다. 특히 [Claude](https://claude.ai) Code Security Review는 코드에서 보안 패턴을 찾는 능력이 뛰어나다. 문제는 그 에이전트 자체가 공격 대상이 된다는 것이다.

### 장점 2: Anthropic의 투명성은 상대적으로 높다

232페이지 시스템 카드는 단순 홍보용 문서가 아니다. 실제로 "프롬프트 인젝션에 hardened되지 않았다"는 취약점을 스스로 명시한 것은 다른 벤더에서 보기 힘든 수준의 투명성이다. 물론 그 사실을 알면서도 제품을 출시했다는 점은 별개의 문제다.

### 장점 3: 패치 속도는 빠르다

취약점 공개 후 세 벤더 모두 빠르게 패치를 배포했다. 수주 또는 수개월이 걸리는 전통적인 엔터프라이즈 소프트웨어 패치 주기와 비교하면 확실히 빠른 대응이다.

### 단점 1: $100 바운티는 보안 연구자 생태계를 약화시킨다

CVSS 9.4 Critical 취약점에 $100은 상징적인 금액이다. 이 금액 수준이 유지되면, 고급 보안 연구자들이 AI 에이전트 취약점 연구를 다른 영역으로 돌릴 가능성이 높다. 결국 피해는 사용자에게 돌아온다.

### 단점 2: CVE 없이 조용한 패치는 기업 보안 팀을 위험에 노출시킨다

많은 기업 보안 팀은 CVE 피드와 CVSS 점수를 기반으로 패치 우선순위를 결정한다. CVE가 발행되지 않으면 이 취약점은 공식 취약점 관리 프로세스에서 누락된다. 실제로 지금도 패치되지 않은 `pull_request_target` 설정으로 AI 에이전트를 운영하는 팀이 있을 것이다.

### 단점 3: 한국어 대응은 전혀 없다

세 벤더 모두 이번 취약점에 대한 한국어 공식 문서나 권고문을 발행하지 않았다. 국내 기업이 이 취약점을 파악하려면 영문 연구 보고서를 직접 읽어야 한다. 한국어 지원 측면에서는 사실상 0점이다.
