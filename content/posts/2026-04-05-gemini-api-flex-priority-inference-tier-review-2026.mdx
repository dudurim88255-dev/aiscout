API 비용이 생각보다 빠르게 불어나는 경험, 한 번쯤 해봤을 거다. 특히 Gemini API를 프로덕션에 붙이면서 "이 요청은 빨라야 하고, 저 요청은 좀 느려도 되는데"라는 고민이 생기기 시작한다.

Google이 그 고민에 응답했다. Flex와 Priority, 두 가지 추론 티어를 Gemini API에 새로 추가했다. 요청 성격에 따라 비용과 응답 보장 수준을 직접 고를 수 있게 된 것이다.


## 30초 요약

- **모델 유형**: Gemini API 추론 서비스 (클라우드 API)
- **핵심 기능**: Flex(비용 절감형)와 Priority(응답 보장형) 두 티어 선택 가능
- **가격**: 티어별 차등 과금 (Priority가 더 높음, 아래 요금제 섹션 참고)
- **대상**: API 기반 앱 개발자, 프로덕션 서비스 운영팀, AI 스타트업
- **한국어 지원**: Gemini 모델 자체의 한국어 지원은 가능하나, 티어 관련 설정 UI·문서는 영문 위주


## 핵심 기능 3가지


### Flex 티어 — 배치성 작업엔 이쪽이 답이다

Flex 티어는 응답 속도보다 비용 효율을 우선시한 모드다. 요청이 몰리는 시간대에는 처리가 지연될 수 있다. 대신 Priority 티어 대비 비용이 눈에 띄게 낮아진다.

데이터 전처리, 야간 배치 요약, 대량 문서 분류처럼 "지금 당장 결과가 필요하지 않은" 작업에 적합하다. 응답 SLA(서비스 수준 계약)를 엄격하게 요구하지 않는 워크플로우라면 Flex가 맞다.

| 항목 | Flex 티어 |
|------|-----------|
| 응답 보장 | 없음 (베스트 에포트) |
| 주요 장점 | 낮은 비용 |
| 적합한 작업 | 배치 처리, 비실시간 분석 |


### Priority 티어 — 사용자가 기다리는 순간엔 이게 필요하다

Priority 티어는 응답 속도와 가용성을 보장하는 모드다. 트래픽이 몰려도 요청 처리 우선순위가 높게 유지된다. 실시간 챗봇, 사용자 인터랙션이 있는 앱에서 진가를 발휘한다.

비용은 Flex보다 높지만, 응답 지연으로 인한 사용자 이탈을 막는 비용과 비교하면 충분히 합리적일 수 있다. 고객 대면 서비스라면 Priority를 기본으로 설정해두는 게 낫다.

| 항목 | Priority 티어 |
|------|--------------|
| 응답 보장 | 높은 우선순위 처리 |
| 주요 장점 | 안정적 응답 속도 |
| 적합한 작업 | 실시간 챗봇, 사용자 인터랙션 |


### 티어 혼용 전략 — 한 서비스 안에서 나눠 쓸 수 있다

Flex와 Priority를 하나의 서비스 안에서 요청 유형별로 분리 적용하는 게 핵심 전략이다. 예를 들어 사용자 대화는 Priority, 백그라운드 로그 요약은 Flex로 라우팅하면 된다.

API 호출 시 파라미터 하나로 티어를 지정하는 방식이라 구현 부담이 크지 않다. 기존 코드베이스를 크게 바꾸지 않고도 비용 최적화가 가능하다는 점이 현실적인 장점이다.
