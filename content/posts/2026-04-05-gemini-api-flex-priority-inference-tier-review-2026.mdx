API 호출 비용이 쌓이는 걸 보면서 "이거 계속 써도 되나" 고민해본 개발자라면, 이번 업데이트가 반가울 거다. Google이 Gemini API에 **Flex**와 **Priority** 두 가지 추론 티어를 공식 도입했다.

결론부터 말하면, 워크로드 특성에 맞게 티어를 나눠 쓸 수 있어서 실용적이다. 비용 구조, 지연 시간 차이, 어떤 상황에 어떤 티어가 맞는지 정리했다.


## 30초 요약

- **모델 유형**: Google Gemini API 추론 서비스 (Flex / Priority 티어)
- **핵심 기능**: 워크로드 우선순위에 따른 비용·속도 조절 가능
- **가격**: Flex 티어는 낮은 단가, Priority 티어는 프리미엄 단가 (플랜 섹션 참고)
- **대상**: Gemini API를 실제 서비스에 연동 중인 개발자 및 AI 스타트업
- **한국어 지원**: API 레벨 기능이므로 별도 한국어 UI 없음. Gemini 모델 자체 한국어 품질은 유지됨


## 핵심 기능 3가지


### 1. Flex 티어 — 비용을 낮추는 대신 지연은 감수

Flex 티어는 Google 인프라의 유휴 자원을 활용해 처리된다. 덕분에 토큰당 단가가 Priority보다 낮다.

트레이드오프는 명확하다. 응답 지연이 불규칙하게 발생할 수 있고, 피크 타임에는 처리가 밀릴 수도 있다. 실시간 응답이 필요 없는 배치 작업, 데이터 전처리, 비동기 파이프라인에 적합하다.

| 항목 | Flex 티어 |
|------|-----------|
| 응답 속도 | 가변적 (보장 없음) |
| 단가 | Priority 대비 낮음 |
| 적합 용도 | 배치 처리, 비실시간 워크로드 |
| 가용성 | 인프라 여유에 따라 변동 |


### 2. Priority 티어 — 응답 속도와 안정성을 돈으로 보장

Priority 티어는 전용 리소스 할당 방식으로 작동한다. 요청이 들어오면 대기열 우선순위가 높아서 빠르고 일관된 응답이 돌아온다.

사용자가 직접 기다리는 프로덕션 환경, 실시간 챗봇, 응답 SLA가 있는 엔터프라이즈 서비스에서 쓸 만하다. 단, 비용은 Flex 대비 높다는 점을 감안해야 한다.

| 항목 | Priority 티어 |
|------|--------------|
| 응답 속도 | 일관적, 낮은 지연 |
| 단가 | Flex 대비 높음 |
| 적합 용도 | 실시간 서비스, 프로덕션 앱 |
| 가용성 | 안정적으로 보장 |


### 3. 티어 선택은 API 파라미터 하나로 끝난다

복잡한 설정 없이 API 호출 시 `inference_tier` 파라미터만 바꾸면 된다. `flex` 또는 `priority` 값 중 선택하면 즉시 적용된다.

이 구조 덕분에 동일한 코드베이스에서 워크로드 유형에 따라 티어를 동적으로 분기할 수 있다. 예를 들어 야간 배치는 Flex, 사용자 요청은 Priority로 라우팅하는 식이다.

```python
# Flex 티어 예시
response = client.models.generate_content(
    model="gemini-2.0-flash",
    contents="분석 작업 내용...",
    config={"inference_tier": "flex"}
)

# Priority 티어 예시
response = client.models.generate_content(
    model="gemini-2.0-flash",
    contents="사용자 실시간 요청...",
    config={"inference_tier": "priority"}
)
```

또한 이 기능은 **API를 직접 쓰는 개발자 전용**이다. Gemini 웹 UI나 일반 사용자 플랜과는 무관하다. API 경험이 없는 분에게는 진입 장벽이 있다.
