API 비용이 쌓이는 속도가 생각보다 빠르다는 걸 실감한 적 있을 거다. 특히 트래픽이 몰리는 시간대엔 레이턴시까지 오르면서 비용과 속도 둘 다 잡기가 어렵다.

구글이 Gemini API에 **Flex**와 **Priority** 두 가지 추론 티어를 새로 공개했다. 결론부터 말하면, 배치성 작업에는 Flex가 확실히 유리하고, 실시간 서비스라면 Priority가 맞다. 어느 쪽을 써야 할지 판단하는 기준을 이 글에서 정리했다.


## 30초 요약

- **모델 유형**: Gemini API 기반 추론 티어 (Flex / Priority)
- **핵심 기능**: 비용과 레이턴시를 티어별로 선택, 동일 모델을 다른 조건으로 호출
- **가격**: Flex는 기존 대비 저렴, Priority는 프리미엄 — 정확한 달러 단가는 아래 요금제 섹션 참고
- **주요 대상**: Gemini API를 실제 서비스나 파이프라인에 연동 중인 개발자·스타트업
- **한국어 지원**: API 레벨 기능이라 UI 언어 구분 없음, 한국어 프롬프트·응답 모두 정상 작동


## 핵심 기능 3가지


### Flex 티어 — 비용을 낮추는 대신 레이턴시를 감수한다

Flex는 구글 인프라의 유휴 컴퓨팅 자원을 활용해 요청을 처리한다. 그만큼 단가가 낮아지는 대신, 응답 시간이 일정하지 않을 수 있다. 배치 데이터 처리, 야간 요약 파이프라인, 비실시간 분류 작업처럼 속도보다 비용이 중요한 상황에 딱 맞는다.

요청이 많은 시간대에는 대기가 생길 수 있다. 서비스 SLA가 빡빡하지 않은 내부 도구나 실험적 파이프라인이라면 Flex를 먼저 고려할 것을 권한다.


### Priority 티어 — 응답 안정성이 필요한 실시간 서비스용

Priority는 전용 리소스를 할당해 요청을 처리한다. 트래픽 피크 타임에도 레이턴시 편차가 작다. 챗봇, 실시간 코드 어시스턴트, 고객 응대 시스템처럼 사용자가 직접 기다리는 환경에 적합하다.

비용은 Flex보다 높지만, 레이턴시 변동으로 인한 UX 손실을 생각하면 합리적인 선택이다. 유료 서비스에서 Gemini를 쓰고 있다면 Priority 티어가 기본 선택지가 될 것이다.


### 동일 모델, 다른 티어 — 코드 한 줄 차이로 전환 가능

Flex와 Priority는 별도 모델이 아니다. 동일한 Gemini 모델을 호출하면서 티어만 파라미터로 지정한다. 기존 코드 구조를 바꾸지 않아도 된다는 뜻이다.

| 항목 | Flex | Priority |
|------|------|----------|
| 리소스 유형 | 유휴 자원 활용 | 전용 리소스 할당 |
| 레이턴시 | 가변적 (피크 시 대기 가능) | 안정적 |
| 권장 용도 | 배치 처리, 비실시간 파이프라인 | 챗봇, 실시간 API 연동 |
| 상대 비용 | 낮음 | 높음 |
| 전환 방법 | 파라미터 변경만으로 가능 | 동일 |

전환 자체가 간단하기 때문에 동일 서비스 안에서 요청 유형별로 티어를 다르게 쓰는 전략도 현실적이다.

## 이런 분께 추천합니다

- **Gemini API를 이미 쓰고 있는 개발자** — 코드 변경 최소화로 비용 최적화 가능
- **배치 데이터 처리 파이프라인을 운영하는 팀** — Flex로 월 API 비용을 의미 있게 줄일 수 있다
- **실시간 챗봇이나 AI 어시스턴트를 서비스하는 스타트업** — Priority로 UX 안정성 확보
- **동일 서비스 내 요청 유형이 다양한 경우** — 요청별로 티어를 나눠 쓰는 혼합 전략에 유리
- **인프라 비용 예측이 중요한 B2B SaaS 팀** — 티어 분리로 비용 구조를 더 정밀하게 설계 가능

아쉬운 점도 솔직히 말한다. 공식 블로그에 구체적인 달러 단가가 나와 있지 않다. "Flex가 저렴하다"는 건 알겠는데, 얼마나 저렴한지 숫자로 확인하기 전까지는 예산 계획을 세우기 어렵다. 실제 단가는 콘솔에 들어가 직접 확인해야 한다.

Flex 티어의 레이턴시 편차가 얼마나 큰지도 공식 문서에 명확히 나와 있지 않다. "가변적"이라는 표현만으로는 실무에서 쓸 수 있는지 판단하기 부족하다. 직접 부하 테스트를 해보고 결정하는 게 맞다.
