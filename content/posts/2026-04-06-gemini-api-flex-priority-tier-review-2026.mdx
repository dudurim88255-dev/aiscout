API 비용 청구서를 보면서 "이걸 계속 써도 되나?" 싶었던 적이 있을 거다. 특히 실험적인 프로젝트나 배치 처리 작업에서 비용이 눈덩이처럼 불어나는 게 문제였다.

Google이 Gemini API에 Flex와 Priority 두 가지 추론 티어를 추가했다. 결론부터 말하면, 용도를 제대로 구분해서 쓸 수 있다면 꽤 실용적인 선택지다. 어떤 티어가 어떤 상황에 맞는지, 수치와 함께 풀어봤다.


## 30초 요약

- **모델 유형**: Gemini API 추론 서비스 (클라우드 기반)
- **핵심 기능**: Flex(저비용·고지연) / Priority(고비용·저지연) 티어 선택
- **가격**: Flex는 기존 대비 최대 50% 저렴, Priority는 기존과 동일하거나 소폭 상승
- **대상**: Gemini API를 사용하는 개발자, 스타트업, 기업 MLOps 팀
- **한국어 지원**: Gemini 모델 자체의 한국어 지원은 유지됨. 티어 설정 UI는 영문


## 핵심 기능 4가지


### 1. Flex 티어 — 비용을 낮추는 대신 응답 시간을 양보한다

Flex 티어는 요청이 몰리지 않는 유휴 시간대에 작업을 처리한다. 덕분에 토큰당 단가가 기존 대비 최대 50% 저렴하다.

단, 응답 지연이 최대 수 분까지 늘어날 수 있다. 실시간 챗봇이나 사용자 인터랙션이 필요한 서비스에는 적합하지 않다.

배치 요약, 대규모 데이터 분류, 야간 자동화 파이프라인처럼 "지금 당장 답이 필요하지 않은" 작업에 최적이다.


### 2. Priority 티어 — 응답 속도와 안정성을 보장한다

Priority 티어는 요청 즉시 처리를 보장하는 SLA 기반 구조다. 응답 시간은 기존 표준 API와 동일하거나 더 빠르게 유지된다.

프로덕션 환경의 실시간 응답, 고객 대면 서비스, 금융·의료처럼 지연이 허용되지 않는 도메인에 적합하다.

비용은 Flex 대비 높지만, 기존 표준 요금과 크게 다르지 않다. 이미 표준 API를 쓰고 있었다면 Priority 티어로 마이그레이션 부담은 거의 없다.


### 3. 티어 선택은 요청 단위로 가능하다

Flex와 Priority는 계정 단위가 아니라 API 요청 단위로 선택할 수 있다. 같은 애플리케이션 안에서도 작업 성격에 따라 티어를 다르게 지정할 수 있다는 뜻이다.

예를 들어, 사용자가 직접 보는 UI 응답은 Priority로, 백그라운드 문서 분석은 Flex로 라우팅하는 구조가 가능하다.

코드 변경은 요청 파라미터에 `inference_tier: "flex"` 또는 `"priority"` 한 줄 추가하는 수준이다.

| 항목 | Flex | Priority |
|------|------|----------|
| 응답 지연 | 수 초 ~ 수 분 | 기존과 동일 (보장) |
| 비용 | 기존 대비 최대 50% 절감 | 기존과 유사 |
| SLA 보장 | 없음 | 있음 |
| 적합 용도 | 배치, 야간 처리, 실험 | 프로덕션, 실시간 서비스 |
| 파라미터 | `inference_tier: "flex"` | `inference_tier: "priority"` |


### 4. 비용 예측 가능성이 높아졌다

기존에는 트래픽 급증 시 비용과 응답 시간이 동시에 불안정해지는 문제가 있었다. 티어 분리로 이 두 변수를 독립적으로 관리할 수 있게 됐다.

Flex 티어 사용량을 늘리면 월 청구액의 분산이 줄어든다. 예산이 한정된 스타트업이나 사이드 프로젝트에서 특히 유용하다.

Google은 향후 Flex 티어에 대한 예산 상한 설정 기능도 추가할 예정이라고 밝혔다.
