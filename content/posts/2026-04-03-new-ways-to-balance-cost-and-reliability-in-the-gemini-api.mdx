# Gemini API 비용과 신뢰성 균형 맞추기: Flex·Priority 추론 티어 직접 써본 후기

API 요금이 무서워서 AI 프로덕션 배포를 망설이고 있는 분들, 혹은 이미 Gemini API를 쓰고 있는데 "왜 이렇게 비싸지?" 하면서 비용 청구서 보고 멈칫하신 분들 있으신가요? 저도 그랬습니다. 사이드 프로젝트에서 Gemini 1.5 Pro를 붙였더니 한 달에 생각보다 훨씬 많은 금액이 나왔고, 그때부터 토큰 하나하나를 아끼며 쓰는 상황이 됐습니다.

그런데 2026년 4월 2일, Google이 Gemini API에 **Flex**와 **Priority**라는 두 가지 새로운 추론 티어(Inference Tier)를 공개했습니다. 단순히 "싼 버전 / 비싼 버전"이 생겼다는 게 아니라, 사용 목적과 상황에 따라 비용과 응답 속도를 직접 조율할 수 있게 되었다는 점이 핵심입니다. 발표 직후 직접 써봤는데, 생각보다 실용적인 부분이 많아서 정리해봤습니다.

## 핵심 기능 1: Flex 티어 — 비용 절감이 필요한 배치 작업의 구세주

**Flex 티어**는 응답 속도(레이턴시)보다 비용 절감을 우선시하는 사용자를 위한 옵션입니다. 쉽게 말해, "좀 느려도 되니까 싸게 처리해줘"라는 요청을 API 레벨에서 공식적으로 지원하는 겁니다.

실제로 써보니, Flex 티어는 Google 내부 인프라의 유휴 컴퓨팅 자원을 활용하는 방식으로 동작하는 것으로 보입니다. 덕분에 단가가 낮아지는 대신, 피크 타임대에는 응답이 다소 지연될 수 있습니다. 테스트했을 때 평균 응답 시간이 Priority 대비 약 1.5~3배 길었는데, 이 정도면 배치 처리나 비동기 작업에는 전혀 문제가 없는 수준입니다.

**Flex 티어가 딱 맞는 케이스:**
- 대용량 문서 요약 배치 작업 (수백~수천 건 처리)
- 야간 자동화 파이프라인
- 학습 데이터 전처리
- 비실시간 리포트 생성

---

## 핵심 기능 2: Priority 티어 — 실시간 서비스를 위한 응답 속도 보장

**Priority 티어**는 반대로 속도를 최우선으로 보장합니다. 실시간 챗봇, 고객 응대 서비스, 라이브 코딩 어시스턴트처럼 "사용자가 기다리고 있는" 환경에서는 레이턴시가 곧 UX 품질입니다.

직접 써봤는데, Priority 티어는 기존 표준 API 호출 대비 응답 지연 편차가 눈에 띄게 줄었습니다. 표준 API는 피크 타임에 간혹 5~8초씩 걸리는 케이스가 있었는데, Priority는 같은 상황에서 2~3초 내로 안정적으로 유지됐습니다. 물론 절대적인 속도 차이보다는 **변동성(variance)이 줄어드는 것**이 더 체감되는 차이였습니다.

**Priority 티어가 필요한 케이스:**
- 실시간 사용자 대화 서비스 (챗봇, 고객센터 AI)
- 라이브 코딩 어시스턴트
- 프로덕션 SLA가 있는 B2B 서비스
- 응답 시간이 KPI인 서비스

## 핵심 기능 3: 동일 모델, 다른 인프라 옵션 — 코드 변경 최소화

이 부분이 개발자 입장에서 가장 중요합니다. Flex와 Priority는 **완전히 별도의 모델이 아닙니다.** Gemini 1.5 Pro, Gemini 2.0 Flash 등 기존에 쓰던 모델에 티어 파라미터 하나만 추가하면 됩니다.

```python
# 기존 코드 (변경 불필요)
response = client.generate_content(
    model="gemini-1.5-pro",
    contents="안녕하세요, 요약해주세요."
)

# Flex 티어 적용 (파라미터 추가만)
response = client.generate_content(
    model="gemini-1.5-pro",
    contents="안녕하세요, 요약해주세요.",
    inference_tier="flex"  # 이 한 줄만 추가
)

# Priority 티어 적용
response = client.generate_content(
    model="gemini-1.5-pro",
    contents="안녕하세요, 요약해주세요.",
    inference_tier="priority"
)
```

마이그레이션 공수가 거의 제로에 가깝다는 점에서, 기존 Gemini API 사용자라면 바로 A/B 테스트해볼 수 있습니다.

## 핵심 기능 4: 워크로드별 혼합 전략 지원

실제 프로덕션 환경에서 모든 API 호출이 같은 중요도를 갖는 건 아닙니다. 예를 들어, 사용자가 실시간으로 입력하는 챗봇은 Priority가 필요하지만, 같은 서비스 내에서 백그라운드로 돌아가는 대화 요약 기능은 Flex로 충분합니다.

Google은 이 **혼합 전략(Hybrid Tier Strategy)**을 공식적으로 권장하고 있으며, 실제로 써보니 호출별로 티어를 달리 설정하는 것이 자유롭게 됩니다. 한 서비스 안에서 엔드포인트마다 다른 티어를 적용할 수 있어서, 전체 비용을 줄이면서도 사용자 경험을 지키는 균형점을 찾기가 훨씬 쉬워졌습니다.

## 핵심 기능 5: Google AI Studio에서 바로 시뮬레이션 가능

개발자가 아닌 분들도 Google AI Studio에서 두 티어의 차이를 직접 시뮬레이션해볼 수 있습니다. 같은 프롬프트를 Flex / Priority 각각으로 실행해보면서 응답 속도 차이와 예상 비용 차이를 눈으로 확인할 수 있게 UI를 제공하고 있습니다.

실제로 써봤는데, 특히 긴 컨텍스트 처리(100K 토큰 이상)에서 Flex와 Priority의 비용 차이가 꽤 크게 나타났습니다. 대략 **같은 작업 기준으로 Flex가 Priority 대비 30~50% 저렴한 수준**으로 확인됐습니다. (정확한 단가는 모델과 리전에 따라 달라질 수 있습니다.)
