OpenAI Privacy Filter 도입을 고민 중이라면, 이 글 하나로 판단을 끝낼 수 있다. 클라우드로 데이터를 보내지 않고도 이름·주소·API 키 같은 민감 정보를 자동으로 걸러낼 수 있다는 게 직접 테스트한 결론이다. 모델 구조, 실제 성능, 라이선스 조건, 아쉬운 점까지 모두 정리했다.

이 모델은 2026년 4월 22일(미국 시각) Hugging Face에 공개됐다. Apache 2.0 라이선스를 채택했기 때문에 상업적으로도 제약 없이 쓸 수 있다. 개인 개발자부터 GDPR 규제를 받는 대기업까지, 실제로 누가 쓰면 유리한지 구체적으로 짚어본다.

## 핵심 요약

- **모델 유형:** 양방향 토큰 분류기 (Bidirectional Token Classifier), Sparse MoE 구조
- **핵심 기능:** 온디바이스 PII 탐지 및 자동 삭제 (이름·주소·이메일·API 키 등 8개 카테고리)
- **가격:** 완전 무료 (Apache 2.0 오픈소스)
- **대상:** 데이터 프라이버시 규제 대응이 필요한 기업, AI 파이프라인 구축 개발자
- **한국어 지원:** 공식 한국어 UI 없음. 모델 자체는 다국어 텍스트 처리 가능하나, 한국어 PII 정확도는 공식 발표 미확인
- **출시일:** 2026년 4월 22일 (Hugging Face 공개 기준)
- **라이선스:** Apache 2.0 (상업적 사용 가능, 소스 공개 의무 없음)
- **파라미터:** 총 15억 (1.5B), 실제 추론 시 활성 파라미터 5천만 (50M)
- **컨텍스트 윈도우:** 128,000 토큰


## OpenAI Privacy Filter란?

OpenAI Privacy Filter는 개인 식별 정보(PII, Personally Identifiable Information)를 클라우드 서버로 전송하기 전에 기기 내부에서 탐지·삭제하는 전용 오픈소스 모델이다. 2026년 4월 22일 Hugging Face에 공개됐으며, Apache 2.0 라이선스가 적용된다.

개발사는 [ChatGPT](https://chat.openai.com)로 잘 알려진 OpenAI다. OpenAI는 초창기 오픈소스 연구 조직으로 출발했으나 [ChatGPT](https://chat.openai.com) 이후 클로즈드 소스 전략으로 방향을 틀었다가, 2025년 gpt-oss 모델 패밀리 공개를 시작으로 다시 오픈소스 생태계에 적극 투자하고 있다. Privacy Filter는 그 흐름의 연장선이다. 앞서 에이전트 오케스트레이션 프레임워크 Symphony도 오픈소스로 공개한 바 있다.

기존 PII 필터 도구와의 결정적 차이는 **처리 방향**과 **실행 위치**다. 전통적인 규칙 기반 필터(정규식, 딕셔너리 매칭)는 문맥을 무시하기 때문에 오탐과 누락이 잦다. Microsoft Presidio나 AWS Comprehend Medical 같은 클라우드 기반 도구는 정확도는 높지만 데이터가 외부 서버를 거친다. Privacy Filter는 양방향 문맥 이해 모델을 기기 안에서 직접 실행하므로, 데이터가 로컬을 벗어나지 않는다.

| 구분 | 규칙 기반 필터 | 클라우드 NLP (AWS/Azure) | OpenAI Privacy Filter |
|---|---|---|---|
| 문맥 이해 | ❌ | ✅ | ✅ |
| 온디바이스 실행 | ✅ | ❌ | ✅ |
| 오픈소스 | 경우에 따라 다름 | ❌ | ✅ (Apache 2.0) |
| 상업적 사용 | 제한적 | 유료 | 무료 |
| 128K 컨텍스트 | ❌ | 제한적 | ✅ |

이 세 가지 조건—문맥 이해, 온디바이스, 상업용 무료—을 동시에 충족하는 도구는 현재 시장에서 Privacy Filter가 사실상 유일하다.


## 핵심 기능 5가지

### 양방향 토큰 분류로 문맥을 앞뒤로 읽는다

일반 LLM은 왼쪽에서 오른쪽으로만 토큰을 예측하는 자동회귀(autoregressive) 방식이다. Privacy Filter는 양방향 토큰 분류기(Bidirectional Token Classifier)로 설계돼 문장을 양쪽 방향에서 동시에 읽는다.

이 차이가 실제로 중요한 이유가 있다. "Alice in Wonderland"에서 "Alice"는 공인이지만, "Alice Johnson, 서울시 종로구 거주"에서 "Alice"는 개인 정보다. 단방향 모델은 이름 뒤에 오는 맥락을 읽지 못해 오탐을 내기 쉽다. 양방향 구조는 앞뒤를 모두 보므로 이런 모호한 케이스를 더 정확하게 처리한다.

- 단방향 모델의 한계: 이름 직후 맥락 미반영 → 문학 작품 속 캐릭터 이름도 PII로 오탐 가능
- 양방향 모델의 이점: "시작-중간-끝-단독(BIOES)" 라벨링으로 엔티티 경계를 논리적으로 추론
- 결과: "John"이 이름의 시작으로 분류되면, "Smith"는 이어지는 성으로 분류됨 (분리 오류 방지)

### Sparse MoE로 1.5B 파라미터를 50M처럼 쓴다

Privacy Filter는 Sparse Mixture-of-Experts(MoE) 아키텍처를 적용했다. 총 파라미터는 15억(1.5B)이지만, 한 번의 추론(forward pass)에서 실제로 활성화되는 파라미터는 5천만(50M)에 불과하다.

이게 왜 중요한지는 실용 관점에서 바로 드러난다. 일반적으로 1.5B짜리 모델을 노트북에서 돌리면 메모리와 속도 양면에서 부담이 크다. 그런데 MoE 구조 덕분에 실제 연산량은 50M 수준이라, 일반 노트북은 물론 웹브라우저(WebGPU 활용) 안에서도 실행할 수 있다. Prime Intellect의 연구 엔지니어 Elie Bakouch는 "트릴리언 스케일 데이터를 저렴하게 필터링하기에 이상적인 구조"라고 평가했다(X, 2026-04-22).

| 항목 | 수치 |
|---|---|
| 전체 파라미터 | 1.5B |
| 활성 파라미터(추론 시) | 50M |
| 컨텍스트 윈도우 | 128,000 토큰 |
| 실행 환경 | 노트북, 브라우저(WebGPU) |

### 128,000 토큰 컨텍스트로 긴 문서를 통째로 처리한다

전통적인 PII 필터는 긴 문서를 여러 청크로 나눠서 처리한다. 이때 페이지 경계에서 엔티티를 놓치거나, 앞 청크에서 언급된 이름을 뒤 청크에서 인식하지 못하는 문제가 생긴다.

Privacy Filter는 128,000 토큰 컨텍스트 윈도우를 지원한다. 일반 법률 계약서나 긴 이메일 스레드 전체를 단일 패스로 처리할 수 있다는 뜻이다. 문서를 조각내지 않아도 되므로 엔티티 추적 오류가 구조적으로 줄어든다.

- 128K 토큰 ≈ 약 10만 단어 분량의 영문 텍스트
- 법률 문서, 의료 기록, 대용량 로그 파일 처리에 특히 유리
- 단일 패스 처리 → 청크 분리로 인한 엔티티 누락 방지

### 8개 PII 카테고리를 자동 감지·삭제한다

Privacy Filter가 현재 지원하는 PII 탐지 카테고리는 아래와 같다.

| 카테고리 | 포함 항목 |
|---|---|
| 개인 이름 | 자연인 이름 |
| 연락처 정보 | 실제 주소, 이메일 주소, 전화번호 |
| 디지털 식별자 | URL, 계정 번호, 날짜 |
| 비밀 정보 | 자격증명, API 키, 비밀번호 |

특히 **Secrets 카테고리**는 기존 PII 필터에서 잘 다루지 않는 영역이다. API 키나 비밀번호가 학습 데이터셋에 포함된 채로 클라우드 모델에 전송되는 사고를 막을 수 있다. 이 카테고리의 존재 자체가 개발자 실수에 의한 자격증명 유출 리스크를 낮춘다.

### Apache 2.0 라이선스로 상업적 제약이 없다

Privacy Filter의 라이선스 선택은 기술 사양 못지않게 중요하다. Apache 2.0은 오픈소스 라이선스 중 가장 허용 범위가 넓은 축에 속한다.

핵심 조건을 정리하면 아래와 같다.

- **상업적 자유:** 이 모델을 내장한 상용 제품을 OpenAI에 로열티 없이 판매 가능
- **커스터마이징 허용:** 의료, 법률, 특정 산업 로그 등 도메인 데이터로 파인튜닝 가능
- **소스 공개 의무 없음:** GPL과 달리 자사 코드베이스를 오픈소스로 공개할 의무가 없음

스타트업이나 ISV(독립 소프트웨어 벤더) 입장에서 이 라이선스는 채택 장벽을 사실상 제로로 낮춘다.


## 직접 써본 솔직한 후기

### 장점 1: 노트북에서 실제로 돌아간다

Hugging Face에서 모델을 내려받아 transformers.js 환경에서 실행해봤다. 일반 개발용 노트북(M2 MacBook Air 기준)에서 짧은 문단 처리 시 체감 지연이 거의 없었다. 정확한 추론 속도 수치는 공식 발표 확인이 필요하지만, 50M 활성 파라미터 덕분에 실용적인 속도가 나온다는 건 확인했다.

브라우저에서 WebGPU로 구동하는 옵션도 실제로 작동했다. 서버 없이 프론트엔드 단에서 PII를 사전 삭제하는 구조를 바로 만들 수 있다. 이건 기존 클라우드 기반 NLP API 방식과 완전히 다른 아키텍처 선택지를 열어준다.

### 장점 2: 128K 컨텍스트 덕분에 긴 문서도 한 번에 처리된다

실제로 15페이지 분량의 영문 계약서를 넣어봤다. 기존에 사용하던 규칙 기반 필터는 청크 경계에서 동일 인물의 이름을 분리 처리하는 오류가 있었다. Privacy Filter는 문서 전체를 단일 패스로 처리하기 때문에 이런 경계 오류가 발생하지 않았다. 특히 법률 문서나 의료 기록처럼 동일 엔티티가 반복 등장하는 긴 문서에서 차이가 명확했다.

### 장점 3: Apache 2.0으로 상업 프로젝트에 바로 연결할 수 있다

GPT-5나 gpt-oss-120B 같은 강력한 추론 모델에 데이터를 보내기 전, Privacy Filter로 로컬에서 먼저 민감 정보를 제거하는 파이프라인을 구성할 수 있다. 이 구조는 GDPR, HIPAA 규정 준수와 최신 AI 성능을 동시에 달성하는 실용적인 해법이다. 라이선스 제약 없이 상용 제품에 바로 내장할 수 있다는 점은 실제 도입 결정 속도를 높인다.

### 아쉬운 점 1: 한국어 PII 정확도는 미검증이다

이 모델의 학습 데이터와 평가 벤치마크는 영어 중심으로 공개됐다. 한국어 이름, 주민등록번호 패턴, 한국 주소 형식에 대한 정확도는 공식 발표에 없다. 직접 한국어 문장으로 테스트했을 때 영어 대비 탐지 정확도가 떨어지는 느낌이 있었다. 한국어 데이터를 주로 다루는 환경이라면 도메인 파인튜닝을 추가로 진행해야 할 가능성이 높다.

### 아쉬운 점 2: 지원 PII 카테고리가 8개로 제한된다

현재 8개 카테고리(이름, 연락처, 디지털 식별자, 비밀 정보)만 지원한다. 의료 분야의 진단명, 금융 분야의 계좌 잔액, 생체 정보 같은 도메인 특화 PII는 기본 모델로는 탐지되지 않는다. 이런 케이스는 파인튜닝이나 추가 후처리 로직을 직접 구현해야 한다. 규제 요건이 복잡한 산업에서는 그대로 쓰기에 한계가 있다.

Privacy Filter 자체는 완전 무료다. 모델을 내려받아 실행하는 데 비용이 들지 않는다. 다만 실제 운영 환경에서는 추론 인프라(GPU 서버 또는 클라우드 인스턴스) 비용이 발생한다. 온프레미스로 구축하면 하드웨어 비용, 클라우드에서 돌리면 인스턴스 비용이 간접 비용이 된다. 50M 활성 파라미터 구조 덕분에 이 간접 비용도 일반 LLM 대비 낮다는 게 장점이다.
